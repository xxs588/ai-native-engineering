# 我越来越觉得，AI 时代先被放大的不是写代码能力，而是产品判断能力

这几年我一个很强的体感是：AI 一提速，很多项目先暴露出来的不是“实现不行”，而是“人根本没想清楚自己要什么”。

过去手写代码慢的时候，很多烂判断会被实现成本拖住。你想法再飘，真要一点点敲页面、改接口、联调、回归，项目天然会被迫收敛一轮。这个过程当然不舒服，但某种程度上也像一种被动熔断。

现在这层被动熔断没了。

页面可以很快生成，接口可以很快搭出来，测试脚手架和文档骨架也能一起长出来。于是原来藏在后面的那些问题，会被更早、更直接地暴露出来：

- 目标其实没定清楚
- 范围其实没收住
- 非目标其实没人写
- 验收其实没人定义
- 优先级其实只是拍脑袋

所以我现在越来越不把 AI 时代的核心矛盾理解成“模型还不够聪明”。很多时候，真正的问题是：**执行突然变便宜了，但判断还停在很粗糙的阶段。**

:::tip[阅读地图]
这篇不讲抽象产品学，只回答三个问题：

1. 为什么 AI 一提速，最先炸的往往是目标、范围和验收
2. `AI Product Manager` 到底是一种什么能力，而不是一个新名词
3. 为什么 Part 2 必须先从 `PRD / 范围裁剪 / 验收口径` 开始，而不是直接开写
:::

# 以前没这么刺眼，不代表它不重要

很多人会觉得这件事像是 AI 时代才突然出现的新问题。

其实不是。

以前这些问题就一直在，只是当时“写出来”本身已经足够贵、足够慢，所以很多错误判断还没来得及放大，就先在实现阶段被拖住了。

现在最危险的变化是，AI 把“做出来”这件事压缩得太快了。于是原来藏在后面的模糊判断，会直接跑到台前来：

- 目标模糊，AI 会自动脑补
- 范围没收，AI 会顺着隐含需求一路扩写
- 没有非目标，AI 会把“顺手做一下”当成美德
- 验收不清，AI 会用“看起来差不多”糊过去
- 优先级没排，AI 会默认把最容易做的先做掉

这就是为什么很多人现在最痛的不是“写不出来”，而是“越写越像样，但越来越偏”。

说得更直白一点：

**代码生成能力像发动机，产品判断像方向盘和刹车。**

发动机越强，方向盘打歪的代价越大，刹车失灵的后果也越严重。

:::important[这篇文章真正想讲的，不是“产品经理变重要了”]
我更想强调的是：AI 把“先判断、再执行”这件事逼成了硬门槛。

这里说的产品判断，不是把文档写得更厚，也不是把话说得更像产品经理，而是先把下面这些边界钉住：

- 这次到底在解决什么问题
- 这一版到底先打哪一仗
- 哪些东西明确先不做
- 做到什么程度算过关
- 什么时候该继续加，什么时候该先收
:::

# 这不是我一个人的错觉，产品入口确实在往前移

如果这只是我个人的体感，那最多算一段牢骚。

但过去这段时间，很多头部产品其实都在往同一个方向收：**把 AI 协作的入口，从“直接写代码”往前挪到规格、任务入口、验收和反馈闭环。**

:::note[三个足够说明趋势的信号（2026-03-15 校验）]
- [Kiro 官方发布文（2025-07-14）](https://kiro.dev/blog/introducing-kiro/) 直接把 `spec-driven development` 放在中心位置。它想解决的不是“聊天框还不够聪明”，而是“需求、设计、任务和约束不能一直靠临场 prompting 漂着走”。
- [GitHub Copilot coding agent 官方文档](https://docs.github.com/en/copilot/how-tos/use-copilot-agents/coding-agent/assign-copilot-to-an-issue) 明确写到：当 issue 被分配给 Copilot 时，它会吃进去标题、描述、已有评论和额外指令；但分配之后新增的评论，它不会自动感知。翻成人话就是，任务入口写得烂，agent 就会很忠诚地把烂需求执行到底。
- [OpenAI《Harness engineering》（2026-02-11）](https://openai.com/index/harness-engineering/) 里已经不只在讲“agent 能不能多写点代码”，而是在讲 `PRODUCT_SENSE.md`、`PLAN.md`、失败反馈和验证闭环。这类资产的核心，已经明显不是纯实现层材料，而是更上游的产品意图和收敛机制。
:::

所以我现在越来越不喜欢把这件事简单理解成“AI 帮忙写代码”。

更准确的说法应该是：**AI 正在逼人把原本含糊的产品判断，提前写成它真的能执行的约束。**

# `AI Product Manager` 到底在判断什么

如果把这件事说得太虚，很容易变成新黑话。所以我更愿意把它拆成几项非常具体的判断。

| 判断项 | 要回答的问题 | 不回答会怎样 | 最小产出 |
|---|---|---|---|
| 问题定义 | 这次到底在解决谁、什么场景、什么痛点 | AI 会自动补完整个世界观 | 一句版本目标 + 目标用户 + 核心场景 |
| 版本边界 | 这一版必须做什么，明确不做什么 | 范围会被“顺手”越滚越大 | `Must / Later / Not now` 清单 |
| 验收口径 | 做到什么程度算完成 | 最后只能靠肉眼和感觉拍脑袋 | 用户路径、边界条件、验收清单 |
| 优先级与节奏 | 哪些先做，哪些依赖前置，哪些能并行 | AI 会优先做最顺手的，不一定是最重要的 | 任务顺序、依赖图、里程碑 |
| 上线判断 | 什么证据足以让这一版进入下一阶段 | Demo 很漂亮，但谁都不敢发 | 测试、回归、日志、风险清单 |

这张表最想强调的一点是：

**产品判断不是“想法更多”，而是“边界更清楚”。**

真正厉害的人，不是每次都能多想十个功能，而是能更早判断：

- 哪一个功能现在值得做
- 哪一个功能应该延后
- 哪一个功能虽然看起来酷，但会把主线带偏

## 它不是替代产品经理，而是在重新分配控制权

我不认为 AI 会让产品经理这个角色凭空消失，也不认为工程师只要会用 agent 就自动拥有完整产品能力。

更准确的变化是：**原来散落在不同角色手里的判断，现在必须更早被结构化。**

以前很多事还能靠口头同步、临场沟通、开发过程中的慢速博弈来补。

现在 AI 一旦接管执行，很多模糊空间会被直接固化成产出。这个时候，谁能更早把目标、范围、验收、优先级写清楚，谁就在事实上掌握了控制权。

所以 `AI Product Manager` 更像一种能力层，而不只是一个 title：

- 工程师需要它，才能不被 AI 带着乱跑
- 产品经理需要它，才能把需求变成可执行约束
- 架构师也需要它，才能知道哪些判断该前置到方案层

# 一个典型翻车方式：AI 不是做不出来，而是会把错目标做得又快又漂亮

看一个很常见的指令：

```text
做一个让异地关系更有陪伴感的产品，先把核心功能做出来。
```

这句话听起来像在定方向，实际上几乎没给边界。

AI 大概率会非常勤奋地开始脑补：

- 聊天
- 提醒
- 打卡
- AI 陪聊
- 情绪分析
- 勋章体系
- 排行榜
- 分享海报

然后两天之后，项目看起来已经很热闹了，但没有人真正回答过：

- 这一版到底验证什么
- 最关键的一条用户路径是什么
- 哪个功能是主线，哪个只是噪声
- 如果只能保住一个体验，应该保哪一个

这就是典型的“方向没定，执行力先溢出”。

我现在更愿意先给 AI 下面这种任务：

```text
先不要写代码，先帮我收敛这个需求。

背景：
- 目标用户：长期异地、需要低打扰但持续连接感的伴侣
- 当前阶段：0.1 版，只验证“每天低成本建立连接”这件事是否成立

这一版必须解决：
- 用户能在 30 秒内完成一次轻量互动
- 互动之后对方能明确感知到“有人在意我”

这一版明确不做：
- 不做公开社区
- 不做复杂成长体系
- 不做 AI 陪聊
- 不做任何需要长期运营才能成立的功能

上线前必须看到的证据：
- 新用户能独立完成核心路径
- 至少一条核心路径可以稳定回归
- 不依赖人工解释，读者能看懂功能为什么存在

请输出：
1. 一句话版本目标
2. 核心用户路径
3. Must / Later / Not now 清单
4. 验收标准
5. 风险与依赖
6. 第一批可执行任务
```

这类指令和“直接写代码”相比，最大的区别不是更长，而是它先把最容易失控的地方卡住了。

AI 依然会做很多事，但它开始围着版本目标转，而不是围着自己的想象力乱扩。

:::tip[一个很实用的纪律]
如果 AI 连版本目标、非目标和验收标准都还说不清，那就不要让它开始大规模写实现。

先收敛，再开工，通常比“边写边想”便宜得多。
:::

# 我现在更认可的最小工作流，不是“先开写”，而是“先定盘”

如果把这一篇压缩成一个最小行动框架，我会建议读者先做这 5 件事：

1. 先写一句版本目标，逼自己回答“这一版到底要证明什么”。
2. 再写 `Must / Later / Not now`，把范围裁剪从脑内感觉变成显式边界。
3. 再写 3 到 5 条验收标准，避免最后只剩“看着差不多”。
4. 然后才让 AI 去拆任务、写 specs、排依赖、起实现。
5. 在实现过程中持续回看这几条，不允许 AI 靠“顺手优化”偷改版本边界。

这套流程听起来一点都不酷，甚至有点像在给高性能跑车装交通规则。

但现实就是这样：AI 时代最贵的失误，已经不是某一行代码写错了，而是方向错了还跑得飞快。

# 这也是为什么 Part 2 不会一上来就写功能

所以这一篇的任务，不是把读者训练成另一种“空谈产品”的人，而是先把一个认知种下去：

**项目主线应该从产品判断开始，而不是从代码生成开始。**

顺着这个逻辑，下一部分才会正式进入 `PRD -> 需求收敛 -> 范围裁剪 -> 验收口径 -> 技术栈 -> 架构 -> 仓库规范` 这条真正能把项目钉稳的路径。
