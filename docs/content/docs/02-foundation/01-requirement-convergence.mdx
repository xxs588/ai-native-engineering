# 拿到 PRD 后，第一件事不是让 AI 开始生成功能

很多团队一拿到 PRD，就会下意识地进入一种很熟悉的节奏：把文档扔给 AI，让它先出页面、先出接口、先出一版“看起来能跑”的东西。表面上看，这很高效，因为屏幕上马上就会有进度；但真正的问题也恰好从这里开始。

AI 最擅长的，是把一个已经收紧的目标快速执行出来。AI 最危险的地方，也在这里：**如果目标本身没有收紧，它会把模糊放大得非常快。**

所以 Part 2 的第一篇，先不谈怎么拆任务，不谈怎么选技术栈，也不谈怎么搭仓库。第一步应该先做一件更枯燥、但决定后面是否会返工的事：

**需求收敛。**

这里说的“收敛”，不是把 PRD 改写成一份更漂亮的文档，也不是把所有未来能力都提前规划得巨细无遗。真正要收的是四样东西：

- 当前版本到底要解决什么
- 当前版本明确不解决什么
- 当前阶段的边界在哪里
- 什么样的结果才算验收通过

这些事不先钉住，后面的 issue 拆分、Spec 细化、代码生成和验证都会一起漂。

# GraphSpec 先要收的，不是单点功能，而是产品定位

GraphSpec 这个项目很容易让人一开始就收歪。

因为它表面上看起来特别像一个功能型产品：扫描仓库、生成依赖关系、渲染成图。于是很多人脑子里的“需求收敛”，就会自动变成这样：

- 要不要先做本地依赖图
- 要不要先做前端画布
- 要不要顺手把 AI 调用也接上
- 要不要先支持编辑 graph

这类问题当然重要，但它们都还排在后面。GraphSpec 在起盘阶段真正先要收紧的，不是某个按钮、某个命令、某个页面，而是它到底是什么。

如果这一层不先讲清楚，后面所有功能判断都会被带偏。因为一个“本地依赖图查看器”和一个“架构上下文控制面”，看起来只差几句话，后面的产品路径其实完全不是一回事。

我现在对 GraphSpec 的判断，比最开始要更明确一些：

**它不能被收敛成单点功能，而要先收敛成一个产品定位。**

这个定位不是“更酷的可视化工具”，也不是“会画图的 OpenSpec”。它更准确的说法应该是：

- **OpenSpec** 偏文本化的 change / spec 流程
- **GraphSpec** 偏结构化的架构上下文控制面

一个负责把 proposal、design、tasks、验收口径这些资产收进文本流程；一个负责把结构关系、影响半径、graph 草稿和空间化浏览收成可查询、可操作的中间层。

这也是为什么 GraphSpec 的收敛，不能停在“做一个本地图看看”上。那样收出来的只是一个功能，不是一个能支撑后续 Skill、CLI、IssueDriven、SpecDriven 继续长下去的产品方向。

# 需求收敛真正要回答的，是这四个问题

很多人把“需求收敛”理解成缩 scope，理解得不算错，但还是太窄了。

在 AI Native 项目里，需求收敛更像是在给未来所有执行设定边界条件。至少要先回答四个问题。

# 第一，当前版本的单一目标是什么

不是“功能列表里有哪些项”，而是这个版本最想回答哪个问题。

GraphSpec 现在的第一阶段，如果收得足够稳，目标应该非常单一：

**我要改这个底层模块，到底会影响谁？**

这个问题一旦立住，很多原来看起来同样重要的东西，优先级就会自动往后排。比如：

- 现在不需要先证明 graph 能不能被 AI 编辑
- 现在不需要先证明多端同步是不是成立
- 现在不需要先证明 OpenSpec bridge 的全部能力

因为这些都不是当前版本要回答的第一问题。

版本目标一旦不是单一的，AI 就很容易在实现时把多个方向一起铺开。最后仓库里看起来功能挺多，但没有一条闭环是真正被做透的。

# 第二，当前版本明确不做什么

这一步比很多人想象中更重要。

因为 AI 不会天然尊重“没说的部分”。相反，只要上下文里出现过某个方向，它就很容易默认那也是这轮应该一起处理的范围。

所以非目标一定要写，而且要写得比目标更狠。

拿 GraphSpec 来说，至少要先把下面这些东西从当前阶段里赶出去：

- 不做云端
- 不做桌面宿主
- 不做 MCP Server
- 不做多语言统一扫描
- 不做复杂语义推断
- 不做 graph 直接改源码

这不是否定这些方向，而是先阻止它们污染当前阶段的上下文。

很多项目一开始会失控，不是因为野心太大，而是因为非目标写得太软。目标总写得很美，非目标只写一句“后续可能支持更多能力”，等于什么都没裁。

:::important
在 AI 协作里，非目标不是礼貌性补充，而是执行边界本身。
:::

# 第三，当前阶段和后续阶段怎么切

需求收敛还有一个经常被忽略的问题：**阶段边界不清楚，版本目标也会慢慢漂。**

尤其是这种主线项目，天然会带出很多后续能力。如果阶段不先切开，团队会很容易把“未来主线”误以为“这轮也该一起做”。

GraphSpec 的例子就特别典型。

它后面明明还有很长的路：

- CLI-first 的只读 graph
- writable graph
- Skill 调 CLI
- 更深的 OpenSpec / IssueDriven / SpecDriven bridge

这些方向都是真的，而且都重要。但它们重要，不代表它们应该在一份总 proposal 里同时细化到可以直接开工。

更稳的方式，是先把阶段路线写粗，把当前阶段写细。

也就是说：

- 后续阶段要存在，但只保留粗粒度判断
- 当前阶段才进入可实施级别

这种写法看起来像是在保守，实际是在保护后面的判断空间。因为你现在根本不可能在项目还没开跑的时候，就把 Skill 协议、graph edit 命令面、OpenSpec bridge 的第一条落地链路一次想周全。

所以需求收敛不是“把所有未来都规划清楚”，而是“把当前阶段和未来阶段清楚分层”。

# 第四，什么结果才算验收通过

很多 PRD 最大的问题，不是没有功能点，而是没有真正可落地的验收口径。

没有验收口径，AI 会很自然地把“做了很多东西”当成“做成了这件事”。可工程上真正值钱的，从来不是堆输出，而是闭环有没有被打通。

GraphSpec 当前阶段如果要验收，我更愿意看的是这种口径：

- 能不能在本地生成稳定的 graph 快照
- 能不能以最小成本查看结构和直接影响范围
- 缺少快照时是不是给出明确错误，而不是偷偷帮用户做别的事
- 是否严格停留在 JS/TS、静态依赖、只读 graph 这个边界里

这些东西一旦写清楚，后面不论谁来接手 issue、谁来和 AI 多轮对话，讨论都会有一个共同锚点。

相反，如果验收只写成“做一个好用的可视化工具”，那等于什么都没定义。因为“好用”既不是边界，也不是验收。

# 为什么总 proposal 不该直接拿来开工

这件事我现在的态度比以前更坚决。

很多人希望有一份“大而全”的 proposal，最好从起盘写到最终形态，然后直接把它扔给 AI，当成总施工图。这个想法在纸面上很诱人，因为它给人一种“项目已经被整体想明白了”的安全感。

但这份安全感大多是假的。

对 GraphSpec 这种项目，更合理的做法反而是：

- 先有一份总纲
- 这份总纲服务于后续 issue 创建
- 真正进入实现前，再让每个关键 issue 长出自己的小规格

我更愿意把这种总纲叫做 `program seed`。它的职责不是替后面所有实现做决定，而是把产品定位、阶段路线、当前范围和后续切分方式先钉住。

接下来才是 `issue seeds` 的事情：哪些问题值得单独开 issue，哪些 issue 需要自己的 `proposal / design / tasks`，哪些只需要按既定边界直接做。

这套思路最值钱的地方在于，它能把“大方向确定”和“小问题持续收敛”接起来。

否则项目会陷入两种常见极端：

- 只有总纲，没有后续薄 issue，最后谁接手都在各自脑补
- 只有零散 issue，没有总纲，最后每个 issue 都在偷偷重定义产品

需求收敛，就是先把这两种极端都拦住。

# IssueDriven 和 SpecDriven，不是二选一

这也是我在这轮 GraphSpec 起盘里感受最深的一点。

很多团队会把 `IssueDriven` 和 `SpecDriven` 说成两个流派，好像选了一个就不用另一个。真到项目里，这种分法其实挺偷懒的。

更真实的关系应该是：

- `IssueDriven` 负责把问题切出来
- `SpecDriven` 负责把切出来的问题收紧

如果没有 issue，spec 很容易空转，写成宏大叙事。  
如果没有 spec，issue 又很容易停在“感觉这里该做点什么”的层面。

所以这两者最好的配合方式，不是互相替代，而是前后咬合。

先有一份总纲型收敛，把产品定位、阶段路线、非目标和验收口径定住；再围绕它不断长出更薄的 issue；必要时，再给某个 issue 写自己的小 proposal。

这时候，需求收敛就不再只是“把文档写完整”，而是在给后面的 issue 拆分制造条件。

# 收敛完成的标志，不是文档更厚了，而是 issue 更好切了

这是我现在判断“这一轮需求收敛到底有没有做成”的一个很实用标准。

如果一轮收敛做完，团队接下来还是不知道该开什么 issue，或者每个人都能从同一份 proposal 里读出完全不同的开工方向，那这轮收敛大概率还没成功。

真正的收敛完成，应该会带来一种非常具体的变化：

- 大家知道当前只做哪个 phase
- 大家知道哪些东西现在不能做
- 大家知道第一批 issue 应该围绕哪条最小闭环展开
- 大家知道后面的 phase 虽然重要，但应该留到后续再细化

对 GraphSpec 来说，这种“更好切 issue”的状态，比仓库里多几个占位页面更值钱。因为它意味着后面的 AI 协作终于不再是无头苍蝇式地乱冲，而是沿着同一条主线稳定往前走。

# 接下来才轮到技术选型和仓库起盘

当版本目标、非目标、阶段边界和验收口径都收紧之后，后面的顺序才成立。

这时候再去讨论技术栈，就不会变成“什么都想兼容”；  
这时候再去搭 workspace，也不会变成“先搭一堆以后可能会用到的空壳”；  
这时候再去拆 issue，才知道哪些是当前 phase，哪些只是路线图里的未来方向。

需求收敛真正买到的，不是几段更漂亮的话，而是后面的每一步都更容易被判断。

对 AI Native 开发来说，这一步尤其值钱。因为后面无论是 Spec、Rules、Skills、SOP，还是代码生成、测试验证、Agent 协作，本质上都在放大当前这一轮收敛的结果。

如果前面收得准，后面放大的是速度。  
如果前面收不准，后面放大的就是返工。
