# 为什么项目刚起步时，最先该长出来的是控制面资产

如果按很多人对“高效开发”的直觉，GraphSpec 现在最应该做的，似乎是赶紧把图渲染出来，或者先把 MCP、AI 调用、自动编辑这种听起来更有想象力的能力做起来。

但我现在反而越来越确认一件事：**AI Native 项目在起盘阶段，最先该长出来的不是功能，而是控制面资产。**

原因并不玄。AI 的执行速度一上来，先被放大的通常不是产能，而是方向错误。人脑一旦边界没收紧，AI 就会把模糊的意图快速翻译成一大坨“看起来像在推进、实际上已经跑偏”的实现。后面再回头清理，不只是删代码那么简单，而是要连同上下文、任务拆解、协作预期一起返工。

所以 GraphSpec 到现在最重要的进展，不是某个页面，也不是某个命令，而是仓库里已经先长出来的这些东西：

- `AGENTS.md`
- README
- OpenSpec change
- 占位 workspace
- 最小命令入口
- test / lint / typecheck 基线

这些东西放在传统项目里很容易被归类成“后面再补的文档和脚手架”。但在 AI 协作环境里，它们其实是同一种资产的不同形态：**给未来所有执行提供稳定入口。**

# 先分清 OpenSpec 和 GraphSpec 各管什么

如果不把这个边界先讲清楚，后面的 Spec、Rules、Skills 都会长歪。

GraphSpec 现在最容易被误解的一点，就是它看起来也带一个 `Spec`，于是很多人会本能地把它理解成“一个更图形化、更酷的 OpenSpec”。这条路从一开始就是错的。

OpenSpec 更擅长承载的是：

- `proposal`
- `design`
- `tasks`
- acceptance
- change history

也就是那些**文本化、线性、面向 change 流程**的资产。

GraphSpec 要补的则是另一个维度：**系统结构怎么被稳定看见、稳定查询、稳定修改，并最终稳定交给 AI 使用。**

它更偏向：

- 架构关系
- 影响半径
- graph draft
- 空间化浏览
- 结构一致性检查

也就是**结构化、空间化、面向架构上下文**的资产。

所以这两者不是谁替代谁，而是两层互补的控制面。我现在对这条链路更愿意用下面这个表达：

`Issue -> Spec -> Graph -> Code -> Validation`

这里的 `Graph` 不是附在文档里的配图，也不是拿来展示给别人看的大框箭头，而是一份真正可查询、可演进、未来还能被 AI 调用的工程资产。

这件事值钱，是因为纯文本 spec 很难稳定回答这些问题：

- 这个底层模块一改，会波及到哪些包、哪些文件、哪些依赖边？
- 当前 proposal 里定义的边界，和仓库里的真实结构是不是已经漂了？
- AI 现在拿到的到底是模块级上下文、文件级上下文，还是某个 blast radius 的局部切片？

这些问题，OpenSpec 不该硬扛。GraphSpec 正是补这一层的。

# 起盘阶段最容易犯的错，是把 proposal 写成总施工图

很多团队说“先写 spec”，其实脑子里想的是另一件事：先把未来几个月所有能力都写成一份巨长 proposal，然后直接扔给 AI 开工。

这种做法在人类单线程时代都经常失真，到了 AI 协作时代只会更危险。因为 AI 很擅长把一份模糊但宏大的文档，执行成一坨边界已经失真的代码，而且写得越快，回滚成本越高。

GraphSpec 这次起盘，反而让我把另一件事想明白了：

**项目早期最需要的 spec，不是“全量实施说明书”，而是“program seed”。**

也就是先写出一份足够稳定的上位路线图，明确几件事：

- 这个项目到底在解决什么，不在解决什么
- 当前阶段只正式规格化哪一段闭环
- 哪些未来方向是主线，但现在只保留粗粒度阶段
- 后续 issue 应该怎么切，哪些问题值得单独长出自己的 proposal / design / tasks

这也是为什么 GraphSpec 当前那份总纲 change，我不会把它理解成一张能直接拿去大推代码的施工单。它更像一个路线图锚点，先把项目的主线、阶段边界和 issue 切分方式钉住。

:::important
总纲型 proposal 最重要的价值，不是“已经把未来都想明白了”，而是“让后面的 issue 更容易长出来，而且不互相打架”。
:::

# IssueDriven 和 SpecDriven，应该是一前一后地接起来

我现在越来越不接受一种偷懒说法：要么 IssueDriven，要么 SpecDriven，好像两者只能二选一。

真实项目里，它们应该是一前一后、一粗一细地接起来。

IssueDriven 解决的是：**先把问题抛出来，并且把问题切得足够薄。**

SpecDriven 解决的是：**在真正动手之前，把这一薄 slice 收敛成一个可执行、可验证、可讨论的边界。**

如果 issue 本身就很大，后面的 spec 一定会跟着变大，然后 AI 会自然滑向“一次性把很多东西都做了”。  
如果没有 spec，issue 又很容易停留在“这里应该做点什么”的层面，最后不同人、不同 Agent 对同一个任务会补出完全不同的脑内版本。

更靠谱的节奏应该是这样的：

- 先用一份总纲 change 把 phase map 和边界钉住
- 再围绕某个具体痛点开一个薄 issue
- 需要时，再让这个 issue 长出自己的 `proposal / design / tasks`
- 只让当前这个 slice 进入实现，不把后面几个阶段一起塞进上下文

这也是为什么 GraphSpec 现在的 `tasks`，不该写成一条线性的“完整施工步骤”，而该写成 issue seeds。它的职责不是指导 AI 一次性把所有阶段做完，而是帮助团队后面持续创建更小、更稳、更适合多轮对话收敛的 issue。

举个非常具体的例子，下面这种 issue 切法就是对的：

```md
Phase 1
- Graph snapshot schema
- JS/TS 静态依赖扫描
- cluster 聚合规则
- graphspec init
- graphspec view
```

每一条都足够具体，也都还没有大到需要把 Phase 2、Phase 3 一起拉进来。

反过来，如果 issue 写成“把 GraphSpec 的 CLI、View、Graph 编辑、Skill、OpenSpec bridge 全做完”，后面所有 spec 都只会一起失真。

# 为什么第一阶段一定是 CLI-first

如果只看“最后是不是要让 AI 调用”这个目标，很多人会本能地想直接上 MCP，或者直接做一个 Agent 写入路径。

但对起盘阶段的 GraphSpec 来说，这不是最稳的路线。

因为 AI 真正要调用的能力，必须先有一个稳定、低噪声、可治理的执行面。这个执行面最自然的形态就是 CLI。

所以它的产品路径我现在反而比最开始更确定：

- Phase 0：仓库骨架、README、`AGENTS.md`、OpenSpec 总纲与占位实现
- Phase 1：只读 graph，先回答“改这个模块会影响谁”
- Phase 2：writable graph，让 graph 不只是查看结果，而是可以承载架构草稿和结构变更
- Phase 3：把稳定 CLI 封成 Skill，让 AI 能通过命令面查询 graph、提交 graph 修改请求
- Phase 4：接进 OpenSpec / IssueDriven / SpecDriven 链路，做一致性检查、结构审查和桥接

这里最关键的不是阶段名，而是顺序。

先有稳定的 CLI，再谈 Skill 调 CLI；先有可读的 graph 资产，再谈 AI 编辑 graph；先有桥，再谈让桥承载交通。否则就会出现一种非常典型的伪进展：产品说明写得一步到位，实际执行面全是临时约定，最后每个 Agent 都在绕过主入口直接碰文件。

# 为什么当前规范故意只把 Phase 1 写到可实施

这件事表面上看像保守，实际上是在给后面留活路。

如果现在就把 `writable graph`、Skill 协议、OpenSpec bridge，甚至未来更深的 AI 接入形态全部细写下来，文档会看起来很完整，但那种完整度大部分是幻觉。因为项目真正的难点根本不在“先想到了多少功能”，而在“每往前走一步，之前的判断会不会被真实问题推翻”。

GraphSpec 当前最需要的，不是未来几个阶段的伪确定性，而是第一条最小闭环：

- `graphspec init [path]`
- `graphspec view`
- `.graphspec/` 轻量快照
- cluster-first 的查看体验
- 只承诺 JS/TS 静态依赖和最小 blast radius 浏览

先把这条链路跑通，后面才有资格继续问下一个问题：graph 到底怎么写、Skill 到底怎么封、OpenSpec 到底先桥哪一段。

所以起盘阶段一个非常关键的工程判断就是：**只把当前 phase 写到“可实施”，把后续 phase 写到“可判断”。**

:::warning
最危险的情况不是文档写得不够多，而是总纲写得太像终局。  
一旦总纲长得像终局，团队就会默认后面很多未决问题都已经被决定，AI 也会跟着把猜测当成既定事实去实现。
:::

# Spec、Rules、Skills、SOP、模板和代码生成，分别该沉淀什么

很多人一提“规范先行”，最后真正落下来的只有几篇文档。这个不够。

AI Native 项目的控制面资产不是单件物品，而是一整套分层：

- **Spec**：负责定义当前 slice 的目标、边界、非目标和验收口径
- **Rules**：负责沉淀长期稳定、跨任务复用的协作约束
- **Skills**：负责封装高复用、低噪声、带明确输入输出的执行能力
- **SOP**：负责把多步流程变成团队可重复遵循的标准路径
- **模板**：负责降低空白页成本，让 issue、proposal、评审和验收不从零开始
- **代码生成**：负责在契约稳定后批量复制结构，而不是在边界没稳时放大猜测

拿 GraphSpec 来说，这几类东西的职责就应该很清楚：

- `AGENTS.md` 属于 Rules，负责约束 AI 协作方式和护栏
- 总纲型 OpenSpec change 属于 Spec，负责把 phase map 和当前边界钉住
- 未来“AI 通过 CLI 查询和编辑 graph”的封装，才是 Skills 该接住的事
- “收到一个需求后先归属 phase，再决定是否要新建 proposal / design / tasks”这类流程，更适合沉淀成 SOP
- GitHub issue 描述模板、proposal seed、评审清单，这些适合做模板
- 真正的代码生成，应该等 graph schema、命令入口、包边界这些底层契约稳定后再大量接入

这里面最值得强调的一点是：**代码生成不应该是起盘阶段的第一动作。**

因为代码生成只能放大已经明确的结构，放大不了没想清楚的边界。边界没稳的时候越早 codegen，后面越容易把猜测固化成仓库事实。

# 项目起盘阶段真正买到的，不是文档数量，而是后续 issue 更好开了

我现在看 GraphSpec 的起盘，有一个很朴素的判断：**如果一份 proposal 不能让后续 issue 更好开，它大概率就还不是一份合格的起盘 proposal。**

因为真正会把项目推着走的，不是“曾经有一份很大的上位文档”，而是后面能不能持续长出一串薄 slice：

- 只讨论 graph snapshot schema
- 只讨论 JS/TS 静态依赖扫描
- 只讨论 cluster 聚合规则
- 只讨论 `graphspec init`
- 只讨论 `graphspec view`
- 只讨论 Skill 调 CLI 的输入输出契约
- 只讨论 OpenSpec bridge 的第一条一致性检查链路

一个个开，一个个收，一个个验证。这才是 IssueDriven 和 SpecDriven 组合起来真正能跑的节奏。

换句话说，项目刚开始时最值钱的，不是功能数量，而是**后面还能不能持续收敛**。

GraphSpec 现在真正完成的第一件事，并不是把 graph 做出来，而是把“怎么一步一步长到那个 graph”这件事先钉住了。

# 接下来才轮到真正开工

如果前面这些底板已经立住，下一步就不是继续空谈蓝图，而是老老实实开第一个薄 issue，进入 Phase 1 的最小闭环。

也就是先把“本地只读 graph”做出来，先回答那个最朴素、但最关键的问题：

**我要改这个底层模块，到底会影响谁？**

这时候再往下写实现，路线就清楚了。因为我们已经知道：

- 这不是一个单纯的本地依赖图查看器
- 它和 OpenSpec 是互补关系，不是替代关系
- 当前阶段先做 CLI-first 的 read-only graph
- 后面的 writable graph、Skill 调 CLI、OpenSpec bridge 都会有自己的 issue 和 spec

这套顺序看起来慢，实际上是在给后面的快打地基。否则所有“高效”都只是提前透支后面的收敛成本。
