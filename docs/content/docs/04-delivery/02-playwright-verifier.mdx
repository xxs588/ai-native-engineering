# 这篇先纠正 Playwright 最常见的低估

很多人提到 Playwright，第一反应还是“E2E 测试框架”。这个理解不能说错，但放在 AI Native 工作流里，它的价值太小了。

真正更值钱的视角是：Playwright 是一种运行时 verifier。

它让页面、流程、截图、DOM 状态、交互路径和失败现场都能被回放、比较、复查，而不是只能靠人肉口述“这里看起来不对”。

# 这篇会把哪几类问题拉回机械验证

这一章会重点讲这些原本最容易口水化的问题，怎么被拉回可验证状态：

- 页面有没有真的按预期加载。
- 某条用户流程有没有在某一步断掉。
- 视觉回归到底是审美争议，还是确定性偏移。
- agent 修完一个问题之后，怎么证明它不是顺手把别的地方带崩了。

这一章最终想收住的判断是：

**Playwright 的真正价值，不只是帮团队多写几条自动化测试，而是把“看起来像对”尽量替换成“可以回放验证”。**
