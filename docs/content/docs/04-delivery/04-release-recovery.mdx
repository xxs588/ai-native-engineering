# 这篇先把“最后一公里”拉回主线

很多 AI 协作到“代码写完”就收工了，但真实项目最容易翻车的地方，恰恰在后面：联调时才发现状态没对齐，验收时才发现口径没统一，发布时才发现回退路径根本没准备。

这不是尾声，而是交付主线本体。

# 这篇会把什么变成默认动作

这一章会把这些动作收成同一条闭环：

- release gate 到底卡什么，不让“差不多能发”混成“真的能发”。
- approval pause 为什么不是拖慢流程，而是在高风险节点帮系统保命。
- incident replay 和 rollback 为什么应该提前设计，而不是事故后临时抱佛脚。
- 每次联调和发布之后，哪些经验必须回写成长期资产，避免下一轮重复付费。

如果这一章写对，读者会看到：真正的交付闭环不是“上线成功一次”，而是系统知道失败时怎么停、怎么退、怎么继续。
