# 这一篇真正要破的，不是“不会用工具”，而是旧脑回路

很多人第一次接触 AI 编程，卡住的地方其实不是不会装工具，也不是不知道哪个模型更强，而是脑子里默认还在沿用旧开发范式：

- 把 AI 当高级自动补全
- 把 Prompt 当万能钥匙
- 把工具选择当单选题
- 把所有问题都理解成“模型还不够聪明”

这也是为什么这一篇叫“破局”。

它真正要破掉的，不是某一个工具使用门槛，而是那套已经开始过时的协作脑回路。

:::important[这一篇的作用]
这一篇不是工具测评，也不是 Prompt 小技巧合集。

它负责先帮读者完成三件事：
- 理解为什么范式已经变了
- 理解为什么工具必须按生态位和岗位去看
- 理解为什么这些能力最后必须被组织成控制面
:::

# 我为什么把这一篇拆成三篇文章

如果这一篇只写一篇大而全的长文，当然也能写。

但那样读者很容易看完只记住一堆名词：模型、IDE、CLI、MCP、Skills、SpecDriven、Harness……最后脑子里还是散的。

所以我把它拆成了三篇，而且每一篇都在回答不同层面的问题。

## 第一篇：先解释“为什么范式变了”

`从"你写 AI 补"到"你说 AI 做"的范式转移` 这一篇，重点不是喊口号，而是回答：

- 为什么 AI 编程不再只是补全
- 为什么模型能力演进会直接改变工具形态
- 为什么交互方式会从 Tab 补全走到任务级委派
- 为什么开发模式会从“写实现”慢慢转向“写约束、写规格、写验收”

所以前面为什么要写那么多模型、交互方式、开发模式，不是在堆信息量，而是在先把一件事钉死：

**如果没有底层能力的跃迁，后面你看到的工具分化、工作流重组、Agentic Coding，都不会成立。**

## 第二篇：再解释“为什么工具不能只看强弱”

`兵器谱：Vibe Coding 工具版图与组合策略` 这一篇，重点也不是评一个“最强工具排行榜”。

它真正想回答的是：

- 为什么 GitHub Copilot 适合做通用底座
- 为什么 Cursor、Trae、Antigravity 这种 Native IDE 各有偏向
- 为什么 CLI 阵营会变成很多人的主战场
- 为什么 Web / 云端工具更适合异步委派、review 或后台执行
- 为什么 `Skills`、MCP、CLI、SpecDriven 这些东西不能混成一锅粥

所以第二篇写那么多不同形态的工具，也不是想告诉读者“都去装一遍”，而是想让读者开始建立一个更像工程系统的意识：

**工具不是玩具箱，而是岗位体系。**

## 第三篇：最后解释“为什么这些东西必须被串成控制面”

`从 Prompt 到 Context 再到 Agent Harness：AI 编程控制面的进化` 这一篇，才是在前两篇的基础上做真正的收束。

它回答的是：

- 为什么 `Prompt` 没死，但已经不再是全部
- 为什么 `Context Engineering` 解决的不是“多喂一点”，而是“信息分发、权重管理和噪声控制”
- 为什么真正决定 agent 能不能稳定交付的，是 `Harness` 这一层 how loop

换句话说：

- 第一篇讲的是**为什么范式变了**
- 第二篇讲的是**现在有哪些岗位和武器**
- 第三篇讲的是**这些东西到底该怎么在脑子里串成一个控制面**

:::tip[这一篇真正的阅读顺序]
先建立“范式为什么变了”的认知，再建立“工具应该怎么分工”的感觉，最后再把这些能力和工具收成一张控制面总图。

这样读者读到第三篇时，才会有那种“哦，原来前面不是散着写”的感觉。
:::

# 读完这一篇，读者应该完成什么心智切换

如果这一篇写得对，读者最后不该只是记住几个新词，而应该开始稳定地这样思考：

- 不再先问“哪个模型最强”，而是先问“这次任务缺的是模型能力、上下文质量，还是执行壳”
- 不再把工具选择理解成单选题，而是开始按岗位、噪声、成本和工作流链路来组合
- 不再把 Prompt 当全部控制面，而是开始区分目标表达、信息治理和执行闭环
- 不再把 AI 失败只归咎于“它太笨”，而是开始追问：目标是不是没说清、Context 是不是喂乱了、Harness 是不是没搭好

到了这里，读者才算真的从“会用 AI”开始进入“会设计 AI Native 工作流”。

# 这也是为什么下一篇会进入“暗礁”

这一篇做的是冷启动和收束，但它还没有开始处理最扎心的工程问题。

因为一旦从“会用”走到“真实落地”，新的问题马上就会冒出来：

- AI 为什么写得越来越快，但 debug 越来越痛苦
- 为什么局部看起来都对，系统一跑就开始边缘条件翻车
- 为什么任务一长，代码和上下文都会开始失控
- 为什么工具越多，反而越需要 guardrail、熔断、回滚和审计

所以这一篇的终点，其实刚好是下一篇《暗礁》的起点。

换句话说，第一篇负责帮读者完成认知破局；第二篇开始让读者直面真实工程里的代价、风险和防线。

:::note[如果这一篇真的起作用了]
读者接下来最该关心的，已经不是“再换一个更强的模型”，而是：

- 任务目标有没有说清楚
- 信息上下文有没有治理好
- 执行 loop 有没有闭环
- 失败之后有没有护栏和反馈系统
:::

# 到这里，第一篇才真正像一个完整的起点

这一篇如果最后只留下四句话，我希望是：

- 模型能力的跃迁，解释了为什么 AI 编程不再只是补全
- 工具生态位的分化，解释了为什么不能继续迷信单工具神话
- `Prompt`、`Context`、`Harness` 的分层，解释了为什么这些能力必须被组织成控制面
- 下一篇开始讲“暗礁”，则是因为真正的工程难点，从来都不是会不会生成，而是怎么避免失控

到这里，`Skills`、MCP、CLI、`Specs`、Cloud tasks 这些词，才不会继续像散装名词一样飘在空中，而会重新落回它们各自的生态位里。
