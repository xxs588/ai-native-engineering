# 你以为你在用 AI，其实你还在”手写代码”

如果你尝试过用 AI 写代码，可能会遇到这样的场景：

让 Cursor 生成了一个带搜索功能的列表页，看起来挺完整的——前端组件、API 调用、分页逻辑都有了。运行一下，报错了。

把错误信息丢给 AI：”帮我修一下这个错误”。AI 改了，再运行，又报新错误。

再丢给 AI：”现在又报这个了”。AI 又改了一版，这次能显示了，但搜索功能不 work。

继续丢：”搜索没反应，帮我看看”。AI 加了防抖，但把分页状态搞乱了。

再丢：”现在翻页有问题”。AI 改了状态管理，但搜索又坏了。

如此循环往复好几轮，token 烧了好几万，你开始骂 AI 太笨。最后代码勉强能跑了，但你看着这一堆补丁摞补丁的代码，心里清楚：**这堆东西自己都看不懂了，下次改需求又得重来。**

**这时候你开始怀疑：AI 到底是提效工具，还是制造麻烦的黑盒？**

# 问题不在 AI，在你的思维方式

很多人以为自己在”用 AI 编程”，但实际上只是把 AI 当成了”高级自动补全”：

- 你还在逐行审查 AI 生成的代码
- 你还在手动复制粘贴到编辑器
- 你还在用传统的断点调试方式
- 你还在把 AI 当成”会写代码的 StackOverflow”

**这不是 AI Native 开发，这是”AI 辅助的传统开发”。**

真正的 AI Native 开发，是一次**思维方式的革命**：

- 从”你写代码，AI 补全”到”你定义目标，AI 编排实现”
- 从”关注代码细节”到”关注系统约束”
- 从”调试代码逻辑”到”调试上下文边界”

这不是工具的升级，而是**编程范式的第三次跨越**。

# 本篇你会学到什么

这一篇的目标，是帮你完成”思想冷启动”——打破旧认知，建立新心智。

**心智切换：从 Coding 到 Orchestration**
- 为什么 Prompt Engineering 已经过时，Context Engineering 才是未来
- 开发者的角色如何从”执行者”转变为”编排者”
- 常见的两大误区：把 Agent 当脚本、把 Prompt 当实现

**工具基线：IDE、Terminal、MCP 与 Skills**
- Cursor、Windsurf、Trae、Claude Code、Kiro 等工具的生态位对比
- 如何根据项目类型（新项目 vs 存量项目）选择合适的工具
- 什么是 MCP（Model Context Protocol），为什么它是 AI 工具链的未来
- Skills 体系：可复用的能力模块，如何通过 Skills 标准化高频任务

**Context Engineering 实战**
- 为什么 AI 总是在你的旧代码里”瞎猜”？如何避免上下文污染
- 手把手教你编写 `.cursorrules` 和 `AGENTS.md`
- 用真实案例展示：同样的 AI 模型，在不同的 Context 约束下，表现天差地别

:::tip[一个关键洞察]
如果你觉得”AI 很笨”，很可能不是模型能力不够，而是你投喂的 Context（上下文）有问题。

就像你雇了一个新员工，但既不给他看项目文档，也不告诉他代码规范，还指望他第一天就能写出完美的代码——这不现实。
:::

准备好了吗？让我们开始这场认知革命。
