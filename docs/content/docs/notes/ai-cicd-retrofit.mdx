# 我先把目标说清楚：我要的不是“能跑”，是“稳定交付”

我接手这个项目时，第一反应不是加新功能，而是先定义“稳定”到底是什么意思。

对这个文档站来说，稳定不等于本地命令跑通。真正的稳定，是下面这些结果同时成立：

- 读者打开页面时，核心内容能正常看到。
- 文章导航、内部链接、页面结构没有明显断裂。
- 前端运行时没有持续报错。
- 移动端不会出现横向溢出这类读者第一眼就能看到的问题。
- 每次改动都能在上线前被验证，而不是上线后靠用户反馈补锅。

我把目标从“命令是否执行成功”改成“交付结果是否可信”，这是后面所有技术选择的出发点。

# 我当时遇到的真实问题，不是一个，是一组

我把不稳定点整理成三类，后面每个设计都是对应解决这三类问题。

- 本地和线上不一致：本地看起来没问题，Netlify Deploy Preview 上才暴露异常。
- 发现时机太晚：问题常常出现在合并后，导致回滚成本变高。
- 排障成本太高：失败时证据不足，只能靠猜，修复效率很差。

如果继续用“本地自测 + 主观判断”的方式，这三类问题只会越来越频繁。

# 我为什么把验收真相放在线上，而不是放在 localhost

我不是否定本地调试，而是明确“最终验收标准”必须和真实交付环境一致。

这个项目的读者访问的是 Netlify 页面，不是我机器上的 `localhost`。所以如果 CI 只验证本地流程，结论天然不可靠。

我最终坚持了一条原则：

- Playwright 的验收默认打 Netlify URL。

这么做的价值很直接：

- 测试环境和交付环境统一，减少“本地绿了线上红了”的错觉。
- 每次失败都能直接对应真实页面，排障更有方向。

# 我为什么把流程拆成三条线，而不是一个大工作流

我早期也想过做一个“全能工作流”，后来放弃了。原因很现实：

- 一条流水线什么都做，失败定位会很慢。
- 审查时也很难看清某次失败到底是基线问题还是线上渲染问题。

所以我把职责强行拆开：

- `Quality Gate`：只负责基础质量闸门。
- `Netlify Preview Guard`：只负责 PR 阶段线上预览巡检。
- `Netlify Production Guard`：只负责合并后线上复查。

这不是形式主义，而是为了把“问题类型”和“修复动作”绑定。

# 第一条线：Quality Gate 解决什么，为什么必须先过

对应文件：`.github/workflows/quality-gate.yml`

我让它只做基线，不掺杂线上巡检：

- `oxlint` 检查改动文件。
- `oxfmt --check` 保证格式一致。
- `typecheck` 防止类型层面的回归。
- `build` 确认静态导出链路可用。
- 大改动但没补 `docs/tests/` 时直接拦截。

我为什么这样做：

- 如果连基线都不稳定，让线上巡检去兜底只会放大 CI 时间和噪音。
- 先把明显不合格改动挡住，后面的 Preview 检查才有意义。

它现在能干什么：

- 快速阻止低质量改动进入主干。
- 对“大改动不补测”给出明确约束，逼着我把测试一并演进。

它不能干什么：

- 它不能证明页面视觉和交互就一定没问题。
- 这部分职责被我刻意留给 Preview/Production 的 Playwright 巡检。

# 第二条线：Netlify Preview Guard 解决什么，为什么是核心

对应文件：`.github/workflows/netlify-preview-guard.yml`

这条线是我最重视的，因为它决定 PR 是否可以被信任。

它的执行路径是：

1. 根据 changed files 推导本次重点路径。
2. 解析 PR 对应的 Netlify Deploy Preview URL。
3. 等待预览可访问。
4. 用 Playwright 对目标路径执行点对点冒烟。
5. 无论成功失败都上传 artifact。

我为什么这样做：

- 我不想每次都全站盲扫，太慢也不经济。
- 我更想让检查跟着改动走，做到“改哪里，查哪里”。
- 失败时必须保留证据，否则我永远在猜。

它现在能干什么：

- 能在 PR 阶段提前拦截读者可见异常。
- 能把失败收敛到具体页面和具体断言。
- 能产出 `playwright-report` 与 `test-results`，让我快速复现。

它为什么和“AI”有关：

- changed files 到 target paths 的推导，本质就是“上下文感知的检查路由”。
- 它不是固定脚本死扫，而是根据改动自动调整检查焦点。
- 配合 webhook，可以把“需要补测”或“失败需修复”交给外部 agent 闭环。

# 第三条线：Netlify Production Guard 解决什么，为什么还要复查

对应文件：`.github/workflows/netlify-production-guard.yml`

我不接受“PR 通过就一劳永逸”这个假设。因为预览环境通过，不代表生产环境 100% 一致。

所以合并到 `main` 后，我保留一轮生产巡检：

- 读取生产 URL。
- 复用同一套 Playwright 冒烟。
- 失败时沉淀 run、artifact 和追踪入口。

我为什么这样做：

- 预览是前置拦截，生产复查是最终兜底。
- 两层检查叠加，才能把“漏网问题”压到最低。

它现在能干什么：

- 合并后快速发现线上回归。
- 避免“主干已经坏了但没人知道”的静默失败。
- 给出可回溯证据，方便我决定回滚还是热修。

# 我踩过的真实坑：`/docs/notes` 404 把预览巡检卡死

我希望这部分写得具体，因为它最能说明“为什么我后来要改规则”。

我新增 `docs/content/docs/notes` 文章后，CI 出现了持续失败：

- Workflow 长时间停在“Wait for preview readiness”。
- Netlify 预览已经部署成功，但状态一直不是 200。

我最初怀疑网络或部署延迟，后来看日志定位到根因：

- 目标路径被推导成 `/docs/notes`。
- 该目录路由在当时预览里是 404。
- 真正可访问的是 `/docs/notes/overview` 和具体文章页。

我做了两个修复动作：

- 修正 `derive-changed-doc-paths`，优先落到具体页面路径。
- 就绪探测改成候选路径机制，不让单个 404 直接卡死整条链路。

我解决了什么：

- 从“预览已好但 CI 误判失败”变成“路径可达就继续执行”。
- 把一个偶发坑变成流程可处理、可复用的规则。

# 我把“能干什么”具体化，不讲空话

这套 AI CI/CD 现在在我手里，能稳定做下面这些事。

- 能把问题前移：在 PR 阶段拦截大多数读者可见异常。
- 能把排障变短：失败后直接看 artifact，不再盲猜。
- 能把检查跟改动绑定：点对点巡检，不靠全站重扫。
- 能把上线风险降下来：合并后还有生产复查。
- 能沉淀长期规则：每次踩坑都能回灌到脚本和工作流。

如果对应到日常开发场景，就是：

- 我改一篇文章，CI 不会无脑全测，而是优先检查相关路径。
- 我改样式导致横向溢出，PR 阶段就会被抓住。
- 我引入了 runtime 错误，Playwright 会从 `console`/`pageerror` 报出来。
- 我改了导航或链接，内部 4xx 会被提前发现。

# 我现在的操作顺序已经固化了

我现在每次改动几乎都按这套节奏：

1. 先判断改动影响面。
2. 必要时同步补测试点。
3. 提 PR，先看 `Quality Gate`。
4. 再看 `Netlify Preview Guard`。
5. 失败就下载 artifact，本地用同一预览 URL 复现。
6. 合并后盯 `Netlify Production Guard`。

我常用复现命令：

```bash
cd docs
E2E_BASE_URL=https://<your-preview>.netlify.app \
E2E_TARGET_PATHS=/docs/<target-path> \
bunx playwright test tests/e2e/production-smoke.spec.ts --config=playwright.config.ts
```

这条命令的核心不是“跑了测试”，而是“复现上下文和 CI 一致”。

# 改造前后对比：我到底把什么变好了

下面这组对比，是我实际工作感受最明显的变化。

| 维度 | 改造前 | 改造后 |
| --- | --- | --- |
| 问题发现时机 | 常在合并后或上线后才发现 | PR 阶段就能被 `Quality Gate` + `Netlify Preview Guard` 拦截 |
| 验收依据 | 以本地 `localhost` 结果为主，和线上可能不一致 | 以 Netlify 预览与生产页面为准，结论更贴近真实交付 |
| 定位方式 | 主要靠人肉复现和猜测 | 有 `playwright-report`、`test-results`、日志链路可追踪 |
| 回归风险 | 依赖个人经验，容易漏掉边角问题 | 有固定检查项和合并后复查，漏网概率显著下降 |
| 迭代信心 | 改动越多越怕回归 | 改动可验证、可拦截，迭代节奏更稳 |

我自己最有体感的变化是两点：

- 以前是“问题出了再解释”，现在是“问题先被系统拦住”。
- 以前是“修了但不确定是否彻底”，现在是“修完可复现、可验证、可回看”。

# 我这次改造最大的价值，不是多了几个 workflow

我最终满意的不是文件数量，而是下面三件事已经落地。

- 问题发现更早：进入主干前就能拦住。
- 问题定位更快：证据链完整。
- 结果判断更准：验收以真实部署页面为准。

这就是我为什么坚持引入 AI CI/CD：

不是让流程更花哨，而是让项目在持续迭代里真正变稳。
