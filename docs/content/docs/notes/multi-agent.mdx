# 先把态度讲明白

我现在对 MultiAgent 的判断很明确：

**它不是让 AI 突然变聪明的魔法，而是把一个已经能跑的 Agent 系统，继续拆成可分工、可并行、可治理的工程系统。**

很多人第一次聊到 MultiAgent，脑子里会自动浮现一个很燃的画面：一群虚拟员工同时开工，规划、写代码、测功能、提 PR，一条龙全包。

听起来当然爽。

但真实工程里，MultiAgent 最先放大的往往不是能力，而是混乱：

- 上下文一旦分发错，错误会在多个 Agent 之间复制。
- 职责一旦边界不清，系统会开始互相踢皮球。
- 验收一旦还是同一套视角，自洽幻觉会被放大，不会被纠正。
- 权限一旦不设护栏，多跑几个 Agent 只是把事故并行化。

:::important[一句话版本]
MultiAgent 解决的不是“更聪明”，而是“把上下文、职责、执行权和验收权拆开”。
:::

这也是为什么我不赞成一上来就把读者扔进 Swarm。

这本书前面要先讲 `Prompt`、`Context`、`Rules`、`Harness`、`MCP`，不是铺垫太长，而是在先搭控制面。控制面，人话就是：**谁派活，谁拿什么上下文，谁有权限，做到什么算完成，出了错谁来兜底。**

如果这个控制面还没成型，MultiAgent 很容易变成另一种黑盒面条代码，只不过这次不是一段函数缠在一起，而是一堆 Agent 缠在一起。

Anthropic 在 2024 年 12 月 19 日发布的 [Building Effective AI Agents](https://www.anthropic.com/engineering/building-effective-agents) 里，核心建议其实非常朴素：先从最简单可行的 agent pattern 开始，只有在复杂度确实带来收益时，再往上加层。LangChain 官方的 [Multi-agent](https://docs.langchain.com/oss/python/langchain/multi-agent) 文档也把话说得很直白：不是每个复杂任务都需要 multi-agent。

# 这本书里的 MultiAgent，出场顺序不是第一章

如果把整本书按“认知冷启动 + 项目生命周期”来走，MultiAgent 的位置大概会是这样：

| 阶段 | 先解决什么 | 这时更像什么形态 |
| ---- | ---------- | ---------------- |
| Part 1 | 范式转变、工具分工、控制面意识 | 先建立判断框架 |
| Part 2 | `SpecDriven`、`Context Engineering`、Rules、Harness | 先把单 Agent 跑稳 |
| Part 3 | 在真实仓库里做前后端开发、测试、联调 | 单 Agent 为主，开始出现明确分工 |
| Part 4 | 任务变长、角色变多、链路变复杂 | 引入 MultiAgent 做拆分和并行 |
| Part 5 | 成本、权限、调度、集群协作 | 再往 Swarm 和治理走 |

这里最关键的不是“MultiAgent 很高级”，而是：

**它通常不是起点，而是单 Agent 闭环已经稳定之后的后手。**

闭环，人话就是：一个 Agent 至少已经能比较稳定地完成“读上下文 -> 动手执行 -> 跑验证 -> 看失败 -> 继续修 -> 交付结果”这条链。

如果这条最小闭环都还没跑通，就急着上 MultiAgent，结果通常不会是“更强”，而是“更贵、更慢、更难 debug”。

:::warning[先别自我感动]
如果系统现在还停留在人肉复制报错、手动喂上下文、没有 trace、没有评测、没有超时控制的阶段，那还不是 MultiAgent 的问题，甚至还没真正进入 Agent Engineering。
:::

# MultiAgent 真正解决的，不是“更聪明”，而是四种工程麻烦

把问题说人话一点，MultiAgent 往往在四种场景里才真的有价值。

| 工程麻烦 | 单 Agent 常见症状 | MultiAgent 的正确拆法 |
| -------- | ----------------- | --------------------- |
| 上下文塞不下 | 一个 Agent 同时背 PRD、代码、日志、测试、部署信息，最后什么都记不住 | 按角色拆上下文，只给当前任务所需材料 |
| 角色互相打架 | 同一个 Agent 既要规划、又要实现、又要 review，目标函数混在一起 | 把规划、实现、验证拆开，避免角色冲突 |
| 延迟越来越高 | 顺序执行一长串任务，前端、后端、测试互相排队 | 把可并行的子任务拆成并发 worker |
| 自洽幻觉放大 | 写代码的 Agent 自己写测试、自己解释结果，最后自己说自己没问题 | 让 reviewer 或 evaluator 拿不同上下文做交叉验收 |

这里面最容易被低估的一点，其实是**验收视角分离**。

很多团队以为 MultiAgent 的价值主要来自并行，但在真实项目里，另一个更大的收益其实是：不要让“同一个脑回路”负责从头到尾的所有判断。

尤其在代码场景里，如果实现者和验证者共享了同样的错误前提，系统非常容易掉进“逻辑自洽 + 测试自洽”的陷阱。代码能跑，测试也绿，但方向一开始就偏了。

这也是为什么 Anthropic 在 2025 年 6 月 13 日公开 [How we built our multi-agent research system](https://www.anthropic.com/engineering/built-multi-agent-research-system) 时，重点并不只是“并发更快”，而是让主 agent 把检索、阅读、汇总这些不同工作拆给专门的 subagent，从而减少单个 agent 被海量上下文压垮。

# 什么时候先别上 MultiAgent

很多“想上 MultiAgent”的冲动，本质上都不是真的需要多 Agent，而是当前系统还有更基础的坑没补。

下面这几种情况，我的建议都偏保守：

- 单 Agent 连最小执行闭环都跑不稳。
- 需求本身还很糊，连交付物和完成条件都说不清。
- 仓库里没有统一的 `Rules`、目录约束和任务边界。
- 没有 trace、日志、回放和失败分类，只能靠聊天记录猜哪里错了。
- 没有超时、步数、Token、权限边界，系统想跑多久就跑多久。

如果这些问题还没解决，多 Agent 往往只会把原本一个地方的混乱，复制成三个地方的混乱。

LangChain 官方文档把动机总结得很实用：multi-agent 常见收益主要来自 `tool calling`、`handoff`、上下文隔离和专门化分工。反过来说，如果这些收益在当前任务里还不成立，那就没必要为了“看起来先进”硬上。

# 一个更像真实项目的分工例子

这本书默认优先用大众化 Web 场景举例，所以拿一个很常见的需求来说：

**给一个已经在线上的 SaaS 后台补“团队成员管理”模块。**

这类任务如果只让一个大 Agent 从头包到尾，常见结果通常是：

- 前端页面很快搭出来了，但权限边界是错的。
- 后端接口补了，但没有沿用现有认证中间件。
- 测试看起来写了，但其实只验证 happy path。
- 上线前才发现涉及数据库迁移、审计日志和邀请链接过期策略。

这时候，多 Agent 的合理拆法大概更像下面这样：

| 角色 | 主要职责 | 核心输出 | 不该碰的事 |
| ---- | -------- | -------- | ---------- |
| Planner | 读 PRD、仓库约束、现有模块，拆任务和依赖 | 子任务清单、风险点、完成条件 | 不直接大改代码 |
| Backend Worker | 补接口、权限校验、审计日志、测试 | API 变更、后端测试结果 | 不负责前端 UI 决策 |
| Frontend Worker | 列表页、邀请弹窗、状态提示、异常态 | 页面代码、交互说明、截图或测试 | 不私自修改服务端契约 |
| Reviewer / Test Worker | 基于 diff、需求和日志做交叉验证 | 风险清单、回归结果、遗漏项 | 不替实现者偷偷补锅 |
| Release Guard | 检查 migration、feature flag、回滚方案、发布说明 | 上线检查单 | 不参与业务实现 |

真正让系统协作起来的，不是“给每个 Agent 起个酷名字”，而是给它们发同一种任务契约。

下面这个任务包，比一大段空泛 Prompt 更有用：

```yaml
role: backend-worker
goal: 为“团队成员管理”补齐邀请、移除、权限校验接口
scope:
  - apps/server/src/modules/team/**
  - apps/server/src/modules/auth/**
  - apps/server/tests/team/**
inputs:
  - docs/prd/team-membership.md
  - openapi/team.yaml
  - existing-rules/auth-and-rbac.md
constraints:
  - 必须复用现有 auth 中间件
  - 不要改动 billing 模块
  - 遇到数据库结构变化必须升级为人工确认
done_when:
  - 邀请、移除、角色变更接口通过测试
  - 审计日志字段完整
  - 返回变更文件清单和剩余风险
escalate_when:
  - 现有 RBAC 模型无法覆盖需求
  - 需要新增 migration
  - 发现前端依赖的契约与 PRD 冲突
```

这个任务包的重点不是格式好看，而是把四件事写死：

- 目标是什么
- 边界在哪里
- 什么算完成
- 什么情况必须升级

没有这四件事，MultiAgent 往往不是协作，而是甩锅。

:::tip[真正该共享的不是所有聊天记录]
大部分时候，subagent 最该拿到的不是完整对话历史，而是压缩后的任务契约、必要上下文和上一步的结构化产物。
这比把几千行聊天记录整段转发过去更稳，也更省 Token。
:::

# 通信层先讲人话，再讲协议

很多 MultiAgent 文章一上来就讲黑板架构、消息总线、协议层、状态同步，读完像在看一份分布式系统期末论文。

这些东西当然重要，但如果不先翻成人话，很容易只剩黑话。

可以先把通信问题拆成三层：

| 场景 | 更像什么 | 常见做法 |
| ---- | -------- | -------- |
| 同一个运行时里的多个角色 | 一个主管给几个同事派活 | `tool call`、`handoff`、共享状态对象 |
| 跨服务、长任务、异步解耦 | 一条消息队列把不同系统串起来 | queue、event bus、job system |
| 跨框架、跨供应商的标准互通 | 不同公司的 Agent 服务开始说同一种话 | MCP、A2A 这类协议 |

到了协议这一层，最容易混淆的是 `MCP` 和 `A2A`。

- [MCP](https://modelcontextprotocol.io/introduction) 更偏向 `agent <-> tool / resource`。官方的比喻很形象，它更像 AI 应用的通用接口层，让模型更稳定地接工具、资源和数据。
- [A2A](https://developers.googleblog.com/en/a2a-a-new-era-of-agent-interoperability/) 更偏向 `agent <-> agent`。Google 在 2025 年 4 月 9 日公开发布这一协议时，重点就是让不同框架、不同供应商的 agent 服务能做任务委派和结果交换。

所以别把它们理解成二选一。

更接近真实世界的情况通常是：

- 一个 agent 先通过 MCP 接数据库、文档、浏览器、代码搜索。
- 然后它再通过 A2A 或自定义任务协议，把某个子任务派给另一个 agent 服务。

这才是“既能接世界，也能接同伴”的完整形态。

# 技术栈别一上来就改全家桶

聊 MultiAgent 时，另一个常见误区是：为了上多 Agent，反向推翻整个技术栈。

这通常没有必要。

如果目标是 Web 产品、管理后台、文档站配套案例，优先看现有技术栈里能不能自然长出来：

| 场景 | 更适合先看什么 | 理由 |
| ---- | -------------- | ---- |
| TypeScript / Next.js / 前后端一体 | [AI SDK Agents](https://ai-sdk.dev/docs/agents/overview) | 适合把 agent loop、tool call、UI streaming 和应用层逻辑接在一起 |
| TypeScript / 需要更强 workflow 与状态编排 | [Mastra](https://mastra.ai/docs) | 更适合把 agent、workflow、memory、工具接入收成框架化结构 |
| Python / 数据流、图编排、研究型任务 | [LangGraph](https://langchain-ai.github.io/langgraph/) | 更适合显式状态图、确定性流转和复杂节点控制 |

这里真正重要的不是“哪一个名字更火”，而是：

**先用离当前项目最近的抽象层。**

如果一个 TypeScript 项目已经有清晰的任务边界、工具集和验证链，那就先别为了 MultiAgent 把全家桶切去 Python。
反过来，如果团队本来就在做数据科学和图编排，那也没必要为了追求前端一体化，硬把所有 agent orchestration 塞回 Web 框架。

# 从 MultiAgent 到 Swarm，中间差的不是数量，而是控制面

很多团队喜欢很快把话题升级到 `Swarm`，好像只要并发多开几个 Agent，事情就立刻有了未来感。

我对这件事的判断也很直接：

**三个终端一起跑，不等于 Swarm。**

真正配得上 `Swarm` 这个词的，通常至少要再补几层东西：

- 有统一调度，而不是靠人工盯着谁先跑谁后跑。
- 有预算和配额，而不是让每个 Agent 自己烧 Token。
- 有状态登记和可回放记录，而不是只剩聊天窗口。
- 有权限边界和人工确认，而不是所有 Agent 默认拿最高权限。
- 有失败隔离和重试策略，而不是一个节点炸了整串全挂。

换句话说，MultiAgent 更像是“分工系统”，而 Swarm 更像是“带治理的分工集群”。

这也是这本书为什么要把 `Rules`、`Harness`、`MCP`、Playwright、成本治理、护栏这些能力，写成全流程伴随能力，而不是等到最后突然空降一章“高级玩法”。

:::note[这本书里的落地边界]
文档工程本身保持 Fumadocs + Next.js 的纯静态导出，也就是 `output: 'export'`。

真正可运行的 MultiAgent 样例，应该放在 `cases/` 里做独立实验工程；`docs/` 负责把控制面、分工、协议和踩坑讲清楚，而不是把长生命周期 agent runtime 硬塞进文档站。
:::

# 如果只保留一个最小落地顺序

如果把这篇文章压成一条可执行的路线，我会建议按这个顺序来：

1. 先把单 Agent 的最小闭环跑稳，至少能读仓库、改代码、跑验证、看失败。
2. 把 `Spec`、`Context`、`Rules`、权限边界整理出来，不要继续靠聊天上下文硬撑。
3. 先拆最有价值的两个角色，通常是“实现者”和“验证者”，别一口气造十个 Agent。
4. 给每个 Agent 建任务契约、超时、步数、升级条件和日志追踪。
5. 当并行、跨服务、跨仓库协作真的开始频繁出现，再往 Swarm、MCP、A2A 和调度层走。

这个顺序看起来不炫，但工程上更稳。

因为 MultiAgent 真正难的，从来不是“再多起几个 agent”，而是让这群 agent 在同一个系统里长期不失控。
